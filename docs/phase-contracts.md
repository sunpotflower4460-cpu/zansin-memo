# Phase Contracts

後続エージェント向けの実装契約。Phase外作業は提案に留め、実装しない。

## Phase 0: Docs Foundation
- Phase名: Docs Foundation
- 目的: コンセプト、MVP、Non-Goals、Phase境界を固定する
- 実装してよい範囲: README/docs/運用ルール文書の整備
- 実装してはいけない範囲: UI実装、依存追加、API連携
- 完了条件:
  - 主要ドキュメントが揃う
  - 用語が統一される
  - 後続エージェントが迷わない
- レビュー観点:
  - コンセプト一貫性
  - MVP/Non-Goals整合性
- テスト/確認観点:
  - ドキュメント参照リンク切れなし（Markdownリンクチェッカーで確認）
  - 欠落ファイルなし
- 次Phaseへ渡すもの:
  - 画面要件
  - データ要件
  - 非目標

## Phase 1: App Shell
- Phase名: App Shell
- 目的: 最小画面遷移の土台を作る
- 実装してよい範囲: Home/Write/Seeds/Detail/Gardenの画面枠、ルーティング、仮データ表示
- 実装してはいけない範囲: 永続化、再浮上ロジック、AI、重いアニメーション
- 完了条件:
  - 主要画面へ遷移できる
  - 低圧トーンのUI骨格である
- レビュー観点:
  - 画面名/導線がdocsと一致
  - 汎用タスク管理UIへ寄っていない
- テスト/確認観点:
  - 主要画面遷移の手動確認
  - モバイル幅での崩れ確認
- 次Phaseへ渡すもの:
  - 画面構造
  - 共通レイアウト/コンポーネント方針

## Phase 2: Seed Data Model
- Phase名: Seed Data Model
- 目的: Seed中心の保存・取得基盤を作る
- 実装してよい範囲: Seed型、CRUD、ローカル保存、Repository抽象
- 実装してはいけない範囲: クラウド同期、認証、課金、AI必須設計
- 完了条件:
  - CRUD成立
  - 再読込で保持
  - 後続Phaseで必要な項目が揃う
- レビュー観点:
  - growthStateが思考育成として扱われている
  - mood/importance/tagsが低圧入力を阻害しない
  - relatedSeedIdsで最小関係表現が可能
- テスト/確認観点:
  - 作成/更新/削除/再読込の確認
  - 空タグ・任意メタ入力での保存確認
- 次Phaseへ渡すもの:
  - 安定したデータアクセス層
  - Seedスキーマ定義

## Phase 3: Write Experience
- Phase名: Write Experience
- 目的: 数秒で書ける低圧入力体験を作る
- 実装してよい範囲: 最小入力フォーム、任意メタ入力（カテゴリ表示/保存は`tags`）、保存成功導線
- 実装してはいけない範囲: 入力必須項目の増加、高機能エディタ化、分類強制
- 完了条件:
  - 一言入力で保存できる
  - タイトルなしでも成立
  - 急かし/評価圧のあるUI文言がない
- レビュー観点:
  - 書くハードルが上がっていない
  - 低圧トーンが守られている
- テスト/確認観点:
  - 最短入力フローの手動確認
  - 入力補助項目未設定での保存確認
- 次Phaseへ渡すもの:
  - 実データ化された種
  - 入力UXの確定パターン

## Phase 4: Seed List & Detail
- Phase名: Seed List & Detail
- 目的: 見返し・再編集の中核体験を成立させる
- 実装してよい範囲: 一覧、検索、フィルタ、ソート、詳細表示、更新
- 実装してはいけない範囲: 高度分析、重い可視化、AI推論依存
- 完了条件:
  - 種を素早く探して再編集できる
  - 状態・メタ情報を更新できる
- レビュー観点:
  - 情報過多で分析画面化していない
  - 実用性と静かなUIのバランスがある
- テスト/確認観点:
  - 検索/フィルタ/ソートの動作確認
  - 詳細更新反映の確認
- 次Phaseへ渡すもの:
  - 再浮上対象選択基盤
  - 変換元データ基盤

## Phase 5: Gentle Resurfacing
- Phase名: Gentle Resurfacing
- 目的: 過去の種とやさしく再会できる導線を作る
- 実装してよい範囲: 今日の種表示、ランダム/スコア再浮上、理由表示
- 実装してはいけない範囲: 通知圧の強い設計、ブラックボックス判定
- 完了条件:
  - 継続利用で再会体験が起きる
  - 提案表示が義務感を生まない
- レビュー観点:
  - 低圧である
  - resurfacingロジックの説明可能性がある
- テスト/確認観点:
  - 候補選定ロジックの単体確認
  - Home表示の手動確認
- 次Phaseへ渡すもの:
  - 再浮上候補と理由
  - 変換導線接続ポイント

## Phase 6: Transform Actions
- Phase名: Transform Actions
- 目的: 種を次のアウトプット形式へ橋渡しする
- 実装してよい範囲: 問い/タスク/記事案/プロジェクト案のテンプレ変換UIと保存
- 実装してはいけない範囲: 生成AI必須化、自動確定、外部API前提
- 完了条件:
  - 4種の変換を選択できる
  - 変換結果を保存できる
- レビュー観点:
  - 「提案」であり押し付けになっていない
  - タスク管理アプリ化していない
- テスト/確認観点:
  - 各変換の保存確認
  - 元Seedへの戻り導線確認
- 次Phaseへ渡すもの:
  - 変換結果データ
  - 関係表示向けリンク情報

## Phase 7: Garden View
- Phase名: Garden View
- 目的: 種・芽・木の関係を軽量に可視化する
- 実装してよい範囲: カード/簡易マップ、状態別視認性、最小関係線表示
- 実装してはいけない範囲: 3D演出、複雑グラフエンジン、過剰アニメーション
- 完了条件:
  - テーマのまとまりを視覚的に把握できる
  - モバイルで破綻しない
- レビュー観点:
  - 見た目優先で重くなっていない
  - 書く/見返す体験を阻害しない
- テスト/確認観点:
  - 関係表示あり/なし両ケース確認
  - パフォーマンスとアクセシビリティ確認
- 次Phaseへ渡すもの:
  - AI補助導入候補点
  - 可視化要件の実測知見

## Phase 8: AI Assist Preparation
- Phase名: AI Assist Preparation
- 目的: MVPを壊さずAI補助を将来追加できる形にする
- 実装してよい範囲: インターフェース定義、フォールバック、セキュリティ方針
- 実装してはいけない範囲: APIキー必須化、認証/課金導入、AI未接続時の機能停止
- 完了条件:
  - AIなしで全機能成立
  - 差し込み点が明確
- レビュー観点:
  - AI主役化していない
  - 非目標（認証/課金/クラウド前提）を破っていない
- テスト/確認観点:
  - AI未設定時フォールバック確認
  - 既存機能回帰確認
- 次Phaseへ渡すもの:
  - 本格AIフェーズ向け技術検証計画
  - リスクとガードレール
